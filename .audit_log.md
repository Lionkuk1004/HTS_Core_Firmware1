## 2026-04-04 — Verify_Heisenbug_Metastable (15단계: 하이젠버그·메타스테이블)

- **프로젝트:** `verify_heisenbug_metastable\Verify_Heisenbug_Metastable.vcxproj`
- **실행 파일:** `verify_heisenbug_metastable\bin\x64\Release\Verify_Heisenbug_Metastable.exe`
- **빌드 / 실행:** Release | x64, 단일 실행 `ALL PASS` (exit 0), wall-clock 약 21s (10M 태그 CAS + 2M 분할 쓰기 샘플 + 10M atomic64 + 50k FSM 라운드)

### 현상 1 — CAS / ABA (IPC_RING 깊이 링 + 스탬프 태그 `head`)

- **강제:** 전용 스레드가 `head`를 (stamp,idx) → (stamp+1,idx+1) → (stamp+2,idx)로 **ABA** 궤적을 연속 CAS로 박아 넣음. 피해 스레드는 슬롯에 페이로드 기록 후 `yield` 지점에서 인터럽트(스레드 선점)에 노출.
- **관측:** 순수 인덱스만 보면 idx가 “돌아온 것처럼” 보일 수 있는 구간이 반복됨. **피해자 CAS 재시도 약 1.216e8회** — 스탬프 상위 32비트가 매 커밋마다 갱신되므로 **stale 기대값 CAS는 실패**하고, **데이터 무결성 위반 0건**으로 1천만 커밋 완료.
- **궤적(방어):** “변하지 않았다”는 착각을 막는 유일한 근거는 **64비트 태그+인덱스 단일 원자 단어**와 `memory_order_acq_rel` 경계. 최적화가 이 CAS를 제거하지 못함은 링크된 실행 바이너리가 재시도 카운터를 현실적으로 폭증시키는 것으로 **간접 입증**됨.

### 현상 2 — 64비트 “찢김” 모사 vs 원자적 방어

- **강제:** `volatile` 32비트 HI/LO 두 번 쓰기 사이에 `release` fence + `yield` — Cortex-M4에서 64비트를 두 스토어로 쪼개는 상황을 소프트웨어로 재현.
- **관측:** 읽기 스레드가 `lo`→fence→`hi` 순으로 스캔할 때 **XOR 불변식(`hi^lo==0xAAAAAAAA`) 파괴가 약 1.60e8회 관측** — “반은 과거·반은 현재” 혼합 시각이 **실제로 숫자로 튀어나옴**(Heisenbug가 숨지 않고 잡힘).
- **궤적(방어):** 동일 프로세스에서 `std::atomic<uint64_t>` 단일 `store(load)` 경로를 1천만 회 스트레스했을 때 **단조 역행(inversion) 0건**. 즉 **펌웨어는 wall/논스/타임스탬프에 분할 스토어 금지 + 원자 로드만 허용** 패턴이 타당함이 데이터로 고정됨.

### 현상 3 — 메타스테이블(Authenticating=PROCESSING) 동시 Disconnect/Timeout

- **강제:** 매 라운드 `IPC_State::PROCESSING`으로 리셋 후, 두 워커가 **동시에** `IDLE`(Disconnect)과 `ERROR_RECOVERY`(Timeout)로 `compare_exchange_weak` 경쟁. `std::barrier(3)`로 메인·워커 동기.
- **관측:** **50,000 라운드** 모두 종료 상태가 `IDLE` 또는 `ERROR_RECOVERY` 중 하나이며 `IPC_Is_Valid_State` 통과. **스핀 상한 2M/라운드 초과·교착 없음** — 중간 비트 글리치 상태에 **머물며 CPU를 씹어먹는** 궤적 없음.
- **궤적(방어):** CFI 테이블(`HTS_IPC_Protocol_Defs.h`)이 허용한 전이만 CAS 목표로 삼고, 실패 시 **재로드 후 재시도**로 결정성 있는 터미널로 수렴.

### 한 줄 요약

15단계 하이젠버그 사냥: **ABA는 태그 CAS로 막았고, 분할 64비트 쓰기는 현장에서 XOR 깨짐으로 증명되었으며, 원자 `uint64_t`와 IPC CAS FSM은 1천만·5만 라운드에서 무결** — 컴파일러 펜스/`volatile`이 “쇼”가 아니라 **실패가 숫자로 드러나는 회로**임을 로그가 뒷받침함.

## 2026-04-04 — Verify_Cosmic_SEU_CFI (16단계: SEU / CFI / 패닉 소각)

- **프로젝트:** `verify_cosmic_seu_cfi\Verify_Cosmic_SEU_CFI.vcxproj`
- **실행 파일:** `verify_cosmic_seu_cfi\bin\x64\Release\Verify_Cosmic_SEU_CFI.exe`
- **빌드 / 실행:** Release | x64, `ALL PASS` (exit 0). 샘플 로그: SEU **5회** 비트 플립으로 CRC+3중 phase 무결성 붕괴 감지 → **try/catch 복구 없이** `Terminate_Session` + `SecureMemory::secureWipe` + `volatile` 재소거 → **CosmicBlob(80B)+키 미러+라이브 enc/mac/iv** 덤프 전바이트 0 검증. CFI: 데이터 영역 디코이 기반 **가짜 함수 포인터 20,000건** 화이트리스트 차단·**실제 jump 미실행**, worst **7µs** (&lt; 1ms). IPC 상태 `0x1F` 다중 비트 글리치는 `IPC_Is_Valid_State` 거부.

### 궤적

- **SEU:** 백그라운드 스레드가 `CosmicBlob` RAM에 **포인터 XOR 비트 플립** 주입. 신뢰 조건: `Crc32Util`(서명 영역 76B) + 3중 `phase==AUTH` — 불일치 시 **즉시 패닉**(세션 연장 시도 없음).
- **소각:** Pimpl 내부는 `Terminate_Session`으로 정리, 호스트 스택 **라이브 키/IV·미러·blob**까지 `secureWipe`+`volatile`로 이중 말아 **덤프 스캔 비0 없음**.
- **PC 타락:** 악성 주소는 **간접 호출 전** 차단; 호스트 보호를 위해 비실행 버퍼로의 실제 점프는 수행하지 않음.
